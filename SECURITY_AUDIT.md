# NsCDE-zh 安全审计报告

本报告对 NsCDE-zh 项目的源代码进行了静态安全分析，涵盖 Shell 脚本、Python 逻辑、C 语言工具及构建脚本。

---

## 1. 审计摘要
*   **总体风险等级**：**低**
*   **主要发现**：
    *   项目整体遵循了安全编码规范，有效防御了符号链接攻击。
    *   发现少量死代码（Python 库中不安全的子进程调用函数），但未被实际使用。
    *   C 语言工具代码简洁，未发现常见的内存管理漏洞。

---

## 2. 详细发现

### 2.1 Python 库中的潜在命令注入风险 (中风险 - 待清理)
*   **位置**：`lib/python/MiscFun.py.in`
*   **描述**：函数 `execWithShell`, `execWithShell1`, `execWithShellThread` 直接使用了 `subprocess.check_output(cmd, shell=True)`。如果未来有新功能调用这些函数并传入未经过滤的用户输入（如自定义调色板名、文件名），将导致命令注入。
*   **状态**：**暂无实际影响**。全局搜索显示项目中暂无代码调用这些函数。
*   **建议**：删除这些未使用的函数，或将其重构为不使用 `shell=True` 的形式（例如将命令拆分为列表传入）。

### 2.2 临时文件处理 (优良)
*   **发现**：脚本（如 `bin/nscde.in`）不使用全局 `/tmp` 目录，而是创建并使用用户家目录下的私有临时目录 `${HOME}/.NsCDE/tmp`。
*   **评估**：这一做法极大地降低了针对临时文件的符号链接攻击风险，是推荐的安全实践。

### 2.3 C 语言工具安全性 (优良)
*   **位置**：`src/colorpicker/colorpicker.c`, `src/XOverrideFontCursor/XOverrideFontCursor.c`
*   **评估**：
    *   代码逻辑紧凑，主要调用 Xlib API。
    *   字符串处理使用了 `strncmp` 等长度限制函数。
    *   未发现 `strcpy`, `sprintf` 或 `gets` 等高危内存操作函数。

### 2.4 构建与权限设置 (优良)
*   **发现**：`Makefile` 中未发现任何 Setuid 或 Setgid 权限设置指令（如 `chmod 4755`）。
*   **评估**：降低了通过二进制漏洞实现权限提升的可能性。

---

## 3. 安全建议

1.  **清理死代码**：尽快移除 `lib/python/MiscFun.py.in` 中的 `execWithShell` 相关函数，以消除潜在的“炸弹”风险。
2.  **增强环境变量清理**：在 `bin/nscde` 启动脚本开头，可以进一步清理不必要的环境变量，以确保 FVWM 及其模块在更纯净的环境下运行。
3.  **输入验证**：如果未来增加用户自定义脚本或主题下载功能，务必对输入的文件名、路径名进行正则过滤，防止目录遍历攻击。
